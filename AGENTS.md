# AGENTS.md

# 프로젝트 목표

SolutionBlogAuto는 Hermes Worker 기반의 블로그 자동화 시스템이다.

최종 목표는 주제를 입력하면 AI가 자료를 수집하고, 글을 작성하며, 이미지를 생성하고, Blogger에 자동 게시하는 것이다.

현재 목표는 완성도보다 **빠르게 동작하는 MVP를 만드는 것**이다.

---

# 기술 스택

- Python 3.13
- uv
- LangGraph
- FastAPI
- Pydantic
- pytest
- Ruff

위 기술 스택은 승인 없이 변경하지 않는다.

새로운 라이브러리는 반드시 ChatGPT의 승인을 받은 후 추가한다.

---

# 프로젝트 원칙

- 항상 실행 가능한 상태를 유지한다.
- 동작하는 코드를 먼저 만든다.
- 과도한 추상화는 하지 않는다.
- 단순한 구현을 우선한다.
- 리팩토링은 MVP 이후 진행한다.

---

# 아키텍처 원칙

다음 항목은 승인 없이 변경하지 않는다.

- 프로젝트 폴더 구조
- 패키지 이름
- 핵심 아키텍처
- Workflow 구조
- Agent 구조

구조 변경이 필요하면 먼저 제안만 한다.

승인 전에는 구현하지 않는다.

---

# 구현 원칙

모든 Public 함수는 Type Hint를 사용한다.

Public Class와 Function에는 Docstring을 작성한다.

클래스는 하나의 책임만 가진다.

불필요한 상속과 추상 클래스는 만들지 않는다.

코드는 이해하기 쉽게 작성한다.

짧고 명확한 코드를 우선한다.

---

# 테스트 원칙

새로운 기능은 가능한 pytest를 함께 작성한다.

기존 테스트는 깨지면 안 된다.

모든 Sprint 종료 시

- 실행 성공
- 테스트 성공

상태를 유지한다.

---

# Sprint 종료 조건

Sprint가 끝나면 반드시 아래 조건을 만족해야 한다.

- 프로젝트 실행 성공
- 테스트 성공
- Git Commit 가능
- 문서 최신화

---

## 개발 방식

구현은 항상 Sprint 단위로 수행한다.

Sprint 범위를 벗어난 기능은 구현하지 않는다.

필요한 경우 TODO만 남기고 구현하지 않는다.

---

## 환경 변수 규칙

- 모든 API Key는 `.env`에서 읽는다.
- API Key를 코드에 하드코딩하지 않는다.
- Provider 선택도 `.env`에서 관리한다.
- Provider를 추가할 때 기존 코드를 수정하지 않고 확장할 수 있도록 구현한다.

---

## 작업 절차

모든 Sprint는 아래 순서를 따른다.

1. 구현 계획 제출
2. ChatGPT 검토
3. 승인
4. 구현
5. 테스트
6. 결과 보고

승인 없이 구현을 시작하지 않는다.

---

## 기존 기능 보호

새 기능을 추가할 때 기존 테스트를 수정하여 통과시키지 않는다.

기존 기능이 깨졌다면 먼저 원인을 수정한다.

기존 테스트는 프로젝트의 계약(Contract)으로 간주한다.
